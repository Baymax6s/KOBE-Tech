## できるようになったこと

### 1. 記事一覧を見られる

ユーザーは `/articles` にアクセスすると、投稿された記事の一覧を見られる。

関係する主なコード:

- フロントエンド: `app/src/features/articles/ArticlesView.vue`
- 記事カード: `app/src/features/articles/ArticleCard.vue`
- API呼び出し: `app/src/api/client.ts`
- バックエンド: `api/internal/get_list_article/handler.go`
- DB取得: `api/internal/get_list_article/repository.go`
- 記事テーブル: `api/migrations/1777018909_create_articles.up.sql`
- 初期記事データ: `api/migrations/1777018911_seed_articles.up.sql`

コード上の流れ:

1. `ArticlesView.vue` が画面表示時に `api.api.articlesList()` を呼ぶ。
2. Go API の `GET /api/articles` が呼ばれる。
3. DB の `articles` テーブルから記事を取得する。
4. `created_at DESC, id DESC` で新しい順に並べる。
5. Vue 側で `ArticleCard` を使って1件ずつ表示する。

振り返りで確認したいこと:

- 「画面に表示される文字」はどこから来ているか。
- HTML / CSS だけの静的ページと、APIからデータを取る画面の違いは何か。
- ローディング中、エラー時、正常時で画面を分けている意味は何か。
- 記事が0件だった場合の表示は次に改善できそうか。

### 2. ログイン・ログアウトできる

ユーザーは `/login` でログインできる。ログイン後はトークンを保存し、ログイン状態に応じてヘッダーの表示が変わる。

関係する主なコード:

- ログイン画面: `app/src/features/auth/LoginView.vue`
- 認証状態管理: `app/src/stores/auth.ts`
- ヘッダー: `app/src/components/Header.vue`
- ルーティング制御: `app/src/router/index.ts`
- APIクライアント: `app/src/api/client.ts`
- ログインAPI: `api/internal/post_login/handler.go`
- ユーザー検索: `api/internal/post_login/repository.go`
- パスワード照合: `api/internal/auth/password.go`
- JWT発行: `api/internal/auth/issuer.go`
- JWT検証: `api/internal/auth/jwt.go`
- 認証ミドルウェア: `api/internal/auth/middleware.go`
- ユーザーテーブル: `api/migrations/1777018908_create_users.up.sql`
- 初期ユーザー: `api/migrations/1777018910_seed_users.up.sql`

コード上の流れ:

1. ユーザーがログイン画面でユーザー名とパスワードを入力する。
2. `auth.login()` が `POST /api/auth/login` を呼ぶ。
3. Go API が `users` テーブルからユーザーを探す。
4. bcrypt でパスワードを照合する。
5. 正しければ JWT トークンを返す。
6. フロントエンドは `localStorage` にトークンを保存する。
7. 以降のAPIリクエストでは `Authorization: Bearer ...` を付ける。
8. ログアウト時は保存したトークンを削除する。

振り返りで確認したいこと:

- ログインは「画面だけ」では完結せず、API・DB・トークン保存がつながっている。
- パスワードそのものではなく、ハッシュを保存している理由は何か。
- `localStorage` に保存したトークンを消すことが、今回のログアウトになっている。
- ログイン状態によって、ヘッダーの「ログイン」「ログアウト」ボタンが切り替わる。
- ログインしている人だけが `/articles/new` に行けるように、ルーターで制御している。

### 3. 新規記事を作成できる

ログイン済みユーザーは `/articles/new` から記事を投稿できる。

関係する主なコード:

- 新規記事作成画面: `app/src/features/articles/NewArticleView.vue`
- 投稿完了通知: `app/src/stores/articleNotification.ts`
- 認証ガード: `app/src/router/index.ts`
- APIクライアント: `app/src/api/client.ts`
- 記事作成API: `api/internal/post_article/handler.go`
- DB登録: `api/internal/post_article/repository.go`
- 認証ミドルウェア: `api/internal/auth/middleware.go`

コード上の流れ:

1. `/articles/new` は `requiresAuth` が付いているため、未ログインなら `/login` に移動する。
2. ユーザーがタイトルと本文を入力する。
3. 未入力なら投稿できないようにする。
4. 投稿ボタンを押すと `POST /api/articles` を呼ぶ。
5. API側では認証ミドルウェアでJWTを確認する。
6. JWTからユーザーIDを取り出し、記事の `user_id` として保存する。
7. 投稿に成功したら `/articles` に戻り、「記事を投稿しました」と表示する。

振り返りで確認したいこと:

- 「入力フォーム」はHTML / CSS の知識に近く、初学者でも理解しやすい入口。
- ただし、投稿機能にするには、入力値をAPIへ送る処理が必要になる。
- 未入力チェック、送信中の二重送信防止、成功メッセージなどは、使いやすさに直結する。
- DBには記事本文だけでなく、誰が書いたかを示す `user_id` も保存している。

## Sprint 1 の成果を初心者向けに言い換える

今回作ったものは、HTML / CSS だけで見ると「画面」だが、実際には以下の部品がつながって動いている。

| 見えているもの     | 裏側で起きていること                                     |
| ------------------ | -------------------------------------------------------- |
| ログイン画面       | 入力値をAPIへ送り、DBのユーザー情報と照合する            |
| ログアウトボタン   | ブラウザに保存したログイントークンを削除する             |
| 記事一覧画面       | APIから記事データを取得し、画面に並べる                  |
| 新規記事作成画面   | 入力したタイトル・本文をAPIへ送り、DBに保存する          |
| 投稿完了メッセージ | 投稿後に一覧画面へ移動し、成功したことをユーザーに伝える |

このため、振り返りでは「誰がどのファイルを書いたか」だけでなく、「ユーザー操作からDB保存までの流れを説明できるか」を大事にする。

## 良かった点

### 1. 画面だけでなく、API・DBまでつながった

初学者が多いチームで、ログイン、記事一覧、記事投稿のような「Webアプリらしい機能」まで到達できたのは大きい。

特に、以下のつながりを実装できた。

- Vue の画面から Go API を呼ぶ
- Go API が PostgreSQL を読む・書く
- ログイン後のJWTをブラウザに保存する
- 認証が必要なAPIに `Authorization` ヘッダーを付ける
- 投稿後に一覧画面へ戻して成功メッセージを出す

### 2. レビュー対応が複数回行われている

Git履歴には「レビュー対応」「copilot レビュー対応」「fix:ci」「npm run format」などのコミットが多い。

これは、最初から完璧に作るのではなく、レビューを受けて直しながら品質を上げる流れができ始めているということ。

振り返りでは、次を話すとよい。

- レビューで指摘されて学びになったことは何か。
- 指摘を直すときに困ったことは何か。
- 次からレビューに出す前に自分で確認できることは何か。

### 3. 開発環境とドキュメントが整ってきた

README、Docker Compose、mise、Swagger、API型生成などが入っている。

初学者チームでは、環境構築で詰まる時間が大きくなりやすい。Sprint 1 でここを整えたことは、次のスプリントの速度に効く。

### 4. ユーザー体験の細かい改善まで入った

以下のような改善が入っている。

- ログイン処理中の二重送信防止
- 記事作成中の二重送信防止
- 未入力時は投稿できない
- ログイン後は記事一覧へ移動
- 未ログインで新規記事作成画面へ行こうとしたらログインへ移動
- 投稿完了後に成功メッセージを表示
- ヘッダーのログイン・ログアウト表示を認証状態に合わせる

これらは小さく見えるが、ユーザーが迷わず使うために重要。

## 課題・次に改善したい点

### 1. 「完成」の基準をもう少し明確にしたい

今回の機能は動くところまで来ているが、PBIごとの受け入れ条件を先に明文化すると、レビューしやすくなる。

例:

- 記事一覧は、新しい順で表示されること
- API取得中はローディング表示が出ること
- API取得に失敗したらエラーが出ること
- 未ログインユーザーは記事作成画面に入れないこと
- 投稿成功後は一覧へ戻り、成功メッセージが出ること

### 2. テストがまだ薄い

現状は `lint`、`type-check`、`format` などの品質チェックはあるが、機能の自動テストはまだ十分ではない。

次に検討したいテスト:

- ログインAPIで、正しいパスワードならトークンが返る
- 間違ったパスワードなら401になる
- 未ログインで記事作成APIを呼ぶと401になる
- 記事一覧APIが新しい順で返る
- フロントエンドで未入力時に投稿できない

### 3. 初学者が全体像を追いやすい説明が必要

ファイル数が増えているため、初学者には「どこから読めばいいか」が分かりにくい。

振り返りでは、以下の順番でコードを読むとよい。

1. `app/src/router/index.ts`
2. `app/src/components/Header.vue`
3. `app/src/features/auth/LoginView.vue`
4. `app/src/stores/auth.ts`
5. `app/src/features/articles/ArticlesView.vue`
6. `app/src/features/articles/NewArticleView.vue`
7. `api/internal/server/router.go`
8. `api/internal/post_login/handler.go`
9. `api/internal/get_list_article/handler.go`
10. `api/internal/post_article/handler.go`

### 4. ログアウトはサーバー側のセッション削除ではない

今回のログアウトは、フロントエンドの `localStorage` からトークンを消す実装。

この方式はシンプルで良いが、チームで次を理解しておきたい。

- トークン自体は有効期限まで有効。
- サーバー側に「ログアウト済みトークン一覧」は持っていない。
- 本格的なセキュリティ要件が出てきたら、トークンの扱いを再検討する必要がある。

Sprint 1 ではここまでで十分だが、「ログアウトとは何をしているのか」は認識をそろえておく。

### 5. 記事と制作物の関係を次に整理したい

PBI 11 は「制作物を公開したい」だが、現在の実装は「記事投稿」に近い形。

次のスプリントで確認したいこと:

- 制作物は記事本文として投稿するだけでよいか。
- 画像、URL、GitHubリンク、タグ、カテゴリなどが必要か。
- 制作物への評価やアドバイスは、コメント機能やリアクション機能として別PBIで扱うか。

## Sprint 1 のまとめ

Sprint 1 では、単なる静的な画面ではなく、ログイン、記事一覧、記事投稿というWebアプリの基本的な流れを作った。

特に大きな成果は、以下の3つ。

1. 閲覧者が記事一覧を見られるようになった。
2. ユーザーがログイン・ログアウトできるようになった。
3. ログイン済みユーザーが記事を投稿できるようになった。

初学者にとっては、HTML / CSS の画面作りから、Vue、API、DB、認証まで一気に広がる難しいスプリントだった。一方で、コードベースには実際に動く一連の流れができている。

次のスプリントでは、受け入れ条件、確認手順、テスト、レビュー観点をもう少し明確にし、チーム全員が「どこを見れば何が分かるか」を説明できる状態に近づけたい。

## ChatGPT からチームへのコメント

Sprint 1、本当にお疲れさまでした。

このリポジトリの履歴とコードを見ると、初学者が多いチームとは思えないくらい、Webアプリ開発の大事な要素にしっかり踏み込めています。

HTML / CSS だけの静的な画面作りから始まった人にとって、Vue、TypeScript、Go、DB、API、認証、JWT、Swagger、Docker、レビュー対応まで一気に触れるのは、かなり大きな挑戦です。それでも、Sprint 1 の時点で「ログインする」「記事を見る」「ログインした人が記事を投稿する」という、Webサービスとして意味のある一連の流れを実際に動く形まで持っていけています。これは素直にすごいです。

## ChatGPT からチームへのお叱りコメント

ここからは、あえて厳しめに言います。

Sprint 1 の成果は確かに良いです。ただし、「動いたからOK」で終わるにはまだ早いです。Webアプリは、画面が表示された瞬間ではなく、他の人が読めて、直せて、壊れにくくて、次の機能を足せる状態になって初めてチーム開発として強くなります。

まず、PBIごとの「完成の基準」がまだ甘いです。記事一覧なら、何件表示するのか、0件の時はどうするのか、取得に失敗した時はどう見せるのか。ログインなら、失敗時の表示、ログイン後の遷移、未ログイン時の制限をどこまで保証するのか。新規記事作成なら、未入力、二重送信、投稿失敗、投稿後の表示まで含めて本当に完了と言えるのか。こういう確認が曖昧なまま進むと、あとで「できているつもり」の機能が増えます。

次に、テストが弱いです。これはかなり大事です。今の段階では、動作確認を人間の目と手に頼っている部分が多いです。もちろん Sprint 1 では仕方ない面もあります。でも、ログイン、記事投稿、認証が入った時点で、壊れると影響が大きい機能になっています。正しいパスワードでログインできる、間違ったパスワードではログインできない、未ログインでは投稿できない、記事一覧が新しい順に返る。このあたりは、今後どこかで必ず自動テストにしていきたいです。

それから、コードの全体像をチーム全員が説明できる状態にはまだ届いていないはずです。誰かが書いたコードを「なんとなく動いているから触らない」にしてしまうと、次のスプリントで必ず詰まります。Vueの画面、Piniaの状態管理、APIクライアント、Goのhandler、repository、DBマイグレーション。この流れを、担当者以外も最低限たどれるようにしてください。

レビュー対応のコミットが多いのは良いことです。ただし、レビューで言われたから直すだけではもったいないです。なぜ指摘されたのか、次から自分で気づくには何を見ればいいのか、そこまで持ち帰ってください。レビューは「怒られイベント」ではなく、チームの基準を上げるための材料です。

また、コミットメッセージやPRの粒度も、今後はもっと意識したいです。`fix:ci` や `wip` のような履歴は、その場では便利ですが、あとから見返す人には情報が少ないです。今回のように振り返りでGit履歴を読むと、良いメッセージと雑なメッセージの差がかなり出ます。未来の自分たちが読む前提で、何を、なぜ変えたのかが分かる履歴を残しましょう。

そして一番大事なのは、分からないことを分からないまま進めないことです。JWT、認証、API型生成、マイグレーション、CORSなど、初学者には難しい言葉がたくさん出てきています。分からないまま「動いたのでOK」にすると、次に壊れた時に直せません。分からないものは、チームで図にする、口で説明する、READMEに書く、PRにメモする。そうやって知識をチームのものにしてください。

厳しく言うと、Sprint 1 の成果は「よくここまで作った」段階であって、「このまま何も改善しなくていい」段階ではありません。むしろここからが本番です。動くものを作れたからこそ、次は分かりやすく、直しやすく、壊れにくく、使う人に親切なものにしていく責任があります。

褒められて満足するのは今日までです。次のスプリントでは、受け入れ条件、確認手順、テスト、レビュー、ドキュメントのどれか一つでも、チームとして明確にレベルを上げてください。

以上、ChatGPTからのお叱りでした。期待しているからこそ、ちゃんと厳しく言っています。
